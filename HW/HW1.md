# การบรรยายของ Kent Beck ในหัวข้อ "Tidy First: A Daily Practice of Empirical Software Design" 
พูดถึง ซอฟต์แวร์ในด้านความสัมพันธ์ระหว่างบุคคลภายในทีมพัฒนาและกับผู้มีส่วนได้ส่วนเสีย ในด้าน Economics และ Human Relationships ในการทำซอฟต์แวร์ ในช่วงต้นพูถึงหนังสือ "Structured Design" โดย Ed Jordan and Larry Constantine ที่พูดถึงแนวคิดพื้นฐานของการเชื่อมโยง (coupling) และความสอดคล้อง (cohesion) ซึ่งเป็นหนังสือในยุค Mainframe/Assembly เบ็คพยายามนำหลักการนี้มาปรับให้เข้ากับยุค Agile และ Continuous Delivery ในปัจจุบัน เกี่ยวกับ กฎการเคลื่อนที่ของนิวตันสำหรับการพัฒนาซอฟต์แวร์ 
  
  ต่อไปมีการพูดถึง loop การทำงาน เราเกิด Idea Behavioral Structure 
วงจรอุบาทว์ Developer รู้สึกผิดที่ทำงานแย่ -> Sponsor รู้สึกว่างานช้าและไม่ได้ดั่งใจ -> Developer ยิ่งรีบเผางานเพื่อให้ทัน -> Code ยิ่งเละ
หากตามที่เบ็คยกตัวอย่างมา เมื่อเราเริ่มมีไอเดีย และเราต้องการเปลี่ยนพฤติกรรมการใช้งาน เราต้องเริ่มคิดถึงว่าโครงสร้างเป็นอย่างไร แต่มันไม่ได้มีแค่การทำงานได้อย่างเดียว 
ยังมีเรื่องเงินทุน สปอนเซอร์ ระยะเวลา การทำงาน ทำได้จริง แต่ใช้เวลาเกิน dev มีสองแบบ 1.ไม่เป็นไร update feature ต่อไปได้ โครงสร้างแย่ลงก็ไม่เป็นไร เราได้ตัง 2.dev ประเภทฉันเป็นคนสร้าง ต้องแก้ให้โครงสร้างดี ซึ่งทำให้สองแบบนี้กลายเป็นความสัมพันธ์ที่ไม่ดี ทั้งต่อสปอนและผู้พัฒนา จนทำใหัผู้พัฒนา เริ่มไม่เคารพงานของตัวเอง 
เบ็คเน้นย้ำว่า นักออกแบบซอฟต์แวร์มีหน้าที่รับผิดชอบไม่เพียงแค่การเขียนโค้ดที่ถูกต้องเท่านั้น แต่ยังรวมถึงการรักษาความสัมพันธ์ที่ดีในทีมด้วย 
  
  สิ่งนี้จึงนำไปสู่แนวคิดหลักของ "Tidy First"  เมื่อเผชิญกับโค้ดที่ยุ่งเหยิงซึ่งต้องการการเปลี่ยนแปลงพฤติกรรม Tidy First คือการเห็นคุณค่าในตัวเองในฐานะโปรแกรมเมอร์และตระหนักว่าความยากลำบากในการเขียนโค้ด มุ่งเน้นไปที่การจัดการกับการเปลี่ยนแปลงที่เกิดขึ้นจริง โดยแยกความแตกต่างจากการออกแบบเชิงคาดการณ์ (การพยายามลดต้นทุนสำหรับการเปลี่ยนแปลงในอนาคตที่เป็นไปได้ทั้งหมด) และการออกแบบเชิงตอบสนอง (การเปลี่ยนแปลงโครงสร้างเฉพาะเมื่อจำเป็นอย่างยิ่งเท่านั้น) 00.00-43.16
